# Proposal: 事件系统重构 + 修仙世界日报

## 问题

当前事件日志系统存在以下问题：

1. **事件无叙事价值** — 战斗事件形如"元婴对决，吸收修为15.3"，没有参与者身份、没有戏剧性、大量同质化重复。晋升和寿尽事件同样扁平，无法引起关注。
2. **修士无身份** — `Cultivator` 仅有数值属性（id/age/cultivation/level/courage），没有名字、没有经历追踪，无法支撑叙事。
3. **事件仅服务于前端展示** — 事件在 Web Worker 内存中产生，通过 postMessage 传递给 React 组件渲染，无持久化、无聚合、无外部消费能力。
4. **纯前端架构** — 整个项目是浏览器 SPA，无后端服务，无法支撑持续运行、定时任务、外部 API 调用等需求。

## 目标

将事件系统从"前端展示日志"升级为"LLM 叙事素材源"，并构建完整的日报生成管线：

1. **后端服务化** — 模拟引擎从 Web Worker 迁移到 Node.js 后端服务，持续运行。前端通过 WebSocket 连接后端，保留实时仪表盘 + 控制能力（暂停/重置/倍速）。
2. **修士身份系统** — 修士达到结丹（Lv2）及以上时分配修仙风格姓名，开始追踪个体履历（击杀数、晋升年份、战斗胜场、巅峰修为等）。
3. **结构化事件系统** — 事件携带完整上下文（参与者身份、事件类型、新闻价值评分），持久化到数据库。
4. **日报生成管线** — 每日定时任务聚合当天模拟事件，按新闻价值筛选分级（S/A/B/C），构建结构化 Prompt 发送至 DeepSeek API 润色生成"修仙世界日报"，通过 QQ 机器人推送。

## 范围

### 包含

- 后端 Node.js 服务搭建（HTTP + WebSocket）
- 模拟引擎后端化（从 `src/engine/worker.ts` 的 Web Worker 模式迁移到 Node.js 进程内运行）
- 修士身份系统（姓名生成器 + Lv2+ 命名 + 履历追踪）
- 事件系统重构（结构化事件定义 + 新闻价值评分 + 数据库持久化）
- 日报生成管线（事件聚合 → Prompt 构建 → DeepSeek API 调用）
- QQ 机器人集成（日报推送）
- 前端改造（Web Worker 通信 → WebSocket 通信，保留所有现有仪表盘功能和控制按钮）

### 不包含

- 修士门派/势力系统（后续迭代）
- 日报的 Web UI 展示（本次仅通过 QQ 机器人推送）
- 模拟规则本身的调整（如 `exponential-death-curve` 变更，独立实施）
- 用户认证/多实例支持

## 技术方案概要

### 架构

```

│  │ (现有引擎    │    │ - 身份分配     │    │ - events      │  │
│  │  直接复用)   │    │ - 价值评分     │    │ - cultivators │  │

│         │                                  └───────┬───────┘  │
│  ┌──────▼───────┐                                  │          │
│  │ WebSocket    │◀─────── 前端控制/实时数据 ────────│          │
│  │ Server       │                                  │          │
│  └──────────────┘                                  │          │
│                                                     │          │
│  ┌─ Cron (每天定时) ────────────────────────────────▼────────┐│
```

### 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 后端运行时 | Node.js | 引擎是 TypeScript，零移植成本 |
| 数据库 | SQLite（via better-sqlite3） | 单实例场景，无需部署数据库服务，嵌入式足够 |
| 前后端通信 | WebSocket（ws 库） | 需要双向实时通信（控制指令 + 数据推送） |
| 身份命名门槛 | Lv2（结丹）及以上 | 低阶修士数量庞大且生命周期短，命名无叙事价值 |
| LLM | DeepSeek API | 用户指定 |
| QQ Bot 协议 | OneBot v11（NapCat/LLOneBot） | Node.js 生态成熟，HTTP API 简单直接 |
| 日报频率 | 每现实天 1 次 | 用户需求 |

### 新闻价值评分体系

| 等级 | 类型 | 示例 | 处理方式 |
|------|------|------|---------|
| S（头条） | 全服首个达到某境界 / 某境界最后一人陨落 / 文明里程碑 | "首位炼虚期修士诞生" | 完整展开，含人物履历 |
| A（要闻） | 高阶战斗（Lv4+）/ 以弱胜强 / 连续跨级晋升 / 知名修士陨落 | "叶凌霄以元婴之身击败化神期宿敌" | 完整展开 |
| B（简讯） | 中阶晋升（Lv2-3）/ 普通高阶战斗 | "本日共 3 名修士晋升元婴" | 聚合为统计数字 |
| C（忽略） | 低阶日常战斗 / 炼气期寿尽 | — | 不进入日报素材 |

## 实施阶段

建议分 4 个阶段逐步交付，每个阶段可独立验证：

### Phase 1：后端服务 + 引擎迁移
- 搭建 Node.js 服务（HTTP + WebSocket）
- 模拟引擎从 Web Worker 迁移到后端进程
- 前端改造为 WebSocket 客户端
- 验收：前端仪表盘功能与现状完全一致，控制按钮正常工作

### Phase 2：身份系统 + 事件重构
- 修士姓名生成器（姓氏库 + 名字用字库，随机组合）
- Lv2+ 修士命名 + 履历追踪（击杀数、战斗胜场、晋升年份、巅峰修为）
- 事件系统重构为结构化格式，携带参与者身份
- 新闻价值评分逻辑
- SQLite 持久化（events 表 + named_cultivators 表）
- 验收：数据库中可查到带身份的结构化事件

### Phase 3：日报生成管线
- 每日事件聚合器（按新闻价值筛选分级 S/A/B/C）
- Prompt 构建器（结构化事件 → DeepSeek Prompt）
- DeepSeek API 集成
- 日报结果存储
- 验收：手动触发可生成一份完整日报文本

### Phase 4：QQ Bot 推送
- OneBot v11 协议对接
- 日报定时推送
- 验收：QQ 群每天收到一份修仙世界日报
